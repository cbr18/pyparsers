# Стратегия парсинга Dongchedi (懂车帝)

В данном документе описаны выявленные ограничения текущей реализации парсера Dongchedi и предложены пути оптимизации для повышения скорости и полноты данных.

## 1. Проблема ограничения страниц (1000-2000 записей)

### Суть ограничения
API Dongchedi (`sh_sku_list`) имеет жесткий лимит на глубину пагинации. Независимо от общего количества машин в базе (например, 100,000+), сервер перестает отдавать данные после 50-й или 100-й страницы (лимит 1000–2000 объектов). 

### Почему это критично
*   **Для Full Sync (Полного парсинга):** Мы никогда не увидим "хвост" базы. Огромный объем старых или недорогих объявлений остается недоступным.
*   **Для Incremental Sync (Инкрементального парсинга):** Обычно инкрементальный парсер смотрит только первые несколько страниц (новые поступления). Однако, если мы хотим убедиться, что не пропустили ни одного изменения в глубине базы, текущий метод пагинации не поможет.

### Решение: Стратегия сегментации (Дробление запросов)
Чтобы "выкачать" всю базу, нужно разбивать запросы так, чтобы каждый результат поиска не превышал 1000 позиций.

**Алгоритм:**
1.  Итерация по **городам** (`city_id`). Большинство городов содержит < 1000 машин.
2.  Для крупных мегаполисов (Пекин, Шанхай), где машин > 1000, вводится дополнительное дробление по **ценовым диапазонам** (`sh_price`):
    *   `0-50,000 CNY`
    *   `50,000-70,000 CNY`
    *   и так далее.
3.  Если и в ценовом диапазоне > 1000 машин, добавляется фильтр по **году выпуска**.

Этот метод позволяет "просканировать" всю базу Dongchedi, оставаясь в рамках лимитов API.

## 2. Оптимизация детального парсинга (Спецификации)

### Текущее состояние
Сейчас для получения полных технических характеристик (60+ полей: двигатель, подвеска, опции) используется **Selenium**. Это:
*   **Медленно:** Загрузка всей страницы с JS занимает 5–15 секунд на машину.
*   **Ресурсоемко:** Требует запуска Chromium в Docker.
*   **Ненадежно:** Зависит от рендеринга и изменений DOM-структуры.

### Предложенное решение: JSON API и `__NEXT_DATA__`
Существует два способа получать те же данные без браузера:

#### Способ А: Внутренний JSON API (Рекомендуемый)
Использование прямого запроса к эндпоинту конфигураций:
`GET https://www.dongchedi.com/motor/pc/car/series/config?series_id={series_id}&car_ids={car_id}`
*   **Плюс:** Возвращает чистый структурированный JSON со всеми ТТХ.
*   **Скорость:** < 500мс на запрос.

#### Способ Б: Извлечение из `__NEXT_DATA__`
Если API требует сложных подписей, можно делать обычный `GET` на страницу параметров и извлекать содержимое тега `<script id="__NEXT_DATA__">`.
*   Внутри этого тега лежит JSON-объект, содержащий все данные, которые Selenium "видит" на странице.
*   Это избавляет от необходимости рендерить страницу.

### Декодирование шрифтов (Font Decryption)
Dongchedi шифрует цифры (цены, мощность) через кастомные шрифты. Для полноценного отказа от Selenium в этот процесс нужно внедрить:
1.  Загрузку `.woff2` файла шрифта из ответа.
2.  Парсинг карты глифов (через библиотеку `fontTools`).
3.  Маппинг зашифрованных Unicode-символов в реальные цифры.

## 3. Выводы и рекомендации
Для качественного развития парсера Dongchedi необходимо:
1.  Отказаться от линейной пагинации в пользу **сегментированного обхода** по городам/ценам.
2.  Заменить Selenium в `fetch_car_specifications` на **прямые JSON-запросы** или парсинг `__NEXT_DATA__`.
3.  Внедрить универсальный обработчик шрифтов для расшифровки числовых данных.
