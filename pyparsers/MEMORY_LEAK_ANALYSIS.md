# Анализ причин роста памяти в pyparsers

## Критические проблемы

### 1. Глобальная HTTP сессия никогда не закрывается
**Файл:** `api/http_client.py:520`
**Проблема:** Глобальный экземпляр `http_client = HTTPClient()` создает сессию `aiohttp.ClientSession`, которая никогда не закрывается. Сессия накапливает соединения, буферы и кэш.

**Код:**
```python
# Создаем глобальный экземпляр HTTP-клиента
http_client = HTTPClient()
```

**Решение:** Добавить закрытие сессии при shutdown приложения или использовать контекстный менеджер.

---

### 2. Словари блокировок накапливаются без очистки
**Файл:** `async_api_server.py:121`
**Проблема:** Словарь `_full_fetch_locks` и `_source_locks` в `TaskService` накапливают блокировки для каждого уникального ключа и никогда не очищаются.

**Код:**
```python
_full_fetch_locks: Dict[str, asyncio.Lock] = {}

def _get_full_fetch_lock(key: str) -> asyncio.Lock:
    lock = _full_fetch_locks.get(key)
    if lock is None:
        lock = asyncio.Lock()
        _full_fetch_locks[key] = lock
    return lock
```

**Решение:** Реализовать LRU кэш или периодическую очистку неиспользуемых блокировок.

---

### 3. Задачи накапливаются в словаре без ограничений
**Файл:** `task_service.py:39`
**Проблема:** Словарь `self.tasks` накапливает все задачи, удаление происходит только при успешной отправке. При ошибках задачи остаются в памяти навсегда.

**Код:**
```python
self.tasks: Dict[str, Task] = {}
```

**Решение:** Добавить TTL для задач, автоматическое удаление старых задач, ограничение размера словаря.

---

### 4. Middleware читает весь response body в память
**Файл:** `async_api_server.py:174`
**Проблема:** При `PERF_ATTACH_BODY=True` middleware читает весь response body в память, что может быть очень большим для больших ответов.

**Код:**
```python
body = await response.body()
json_body = json.loads(body)
```

**Решение:** Ограничить размер body для обработки или использовать streaming.

---

### 5. Selenium WebDriver создает временные директории, которые не удаляются
**Файлы:** 
- `api/dongchedi/parser.py:339`
- `api/che168/parser.py:98, 500`

**Проблема:** При каждом вызове `fetch_car_detail` создается временная директория через `tempfile.mkdtemp()`, но она никогда не удаляется.

**Код:**
```python
import tempfile
temp_dir = tempfile.mkdtemp()
chrome_options.add_argument(f"--user-data-dir={temp_dir}")
```

**Решение:** Использовать `shutil.rmtree()` в `finally` блоке или контекстный менеджер.

---

### 6. Selenium WebDriver может не закрываться при исключениях
**Файлы:** 
- `api/dongchedi/parser.py:346-374`
- `api/che168/parser.py:172-268`

**Проблема:** Если происходит исключение до блока `finally`, драйвер может остаться в памяти. Также `driver.quit()` может не освободить все ресурсы.

**Решение:** Использовать контекстный менеджер или гарантировать закрытие в `finally`.

---

### 7. BeautifulSoup объекты удерживают большие HTML документы
**Файлы:** 
- `api/dongchedi/parser.py:371`
- `api/che168/parser.py:205, 525`

**Проблема:** BeautifulSoup парсит весь HTML в память и создает дерево объектов, которое может быть очень большим. Эти объекты могут не освобождаться сразу.

**Код:**
```python
soup = BeautifulSoup(page_source, 'html.parser')
```

**Решение:** Явно удалять soup объекты после использования, использовать более легковесные парсеры.

---

### 8. Большие списки машин накапливаются в памяти
**Файлы:**
- `async_api_server.py:397, 814`
- `task_service.py:202, 286`

**Проблема:** При парсинге всех машин создаются большие списки (`all_cars`, `data`), которые могут содержать тысячи объектов. Даже с `MemoryOptimizedList` и `gc.collect()`, память может не освобождаться полностью из-за циклических ссылок.

**Решение:** Использовать генераторы, батчинг, streaming.

---

### 9. Глобальные экземпляры парсеров удерживают ресурсы
**Файл:** `async_api_server.py:200-201`
**Проблема:** Глобальные экземпляры парсеров создаются при старте и живут все время работы приложения, удерживая возможные внутренние кэши и ресурсы.

**Код:**
```python
dongchedi_parser_instance = DongchediParser()
che168_parser_instance = Che168Parser()
```

**Решение:** Использовать пул парсеров или создавать по требованию.

---

### 10. TaskService сессия может не закрываться при ошибках
**Файл:** `task_service.py:43-47`
**Проблема:** Сессия создается при первом использовании, но может не закрыться при ошибках в `send_to_datahub`.

**Решение:** Использовать контекстный менеджер или гарантировать закрытие.

---

### 11. Response объекты могут удерживать большие данные
**Файл:** `api/http_client.py:266-274`
**Проблема:** Метод `_request` читает весь текст ответа и JSON в память, даже если данные большие.

**Код:**
```python
text = await response.text()
json_response = await response.json()
```

**Решение:** Использовать streaming для больших ответов.

---

### 12. Словари `seen_ids` накапливаются без ограничений
**Файлы:**
- `async_api_server.py:398, 815`
- `task_service.py:238`

**Проблема:** Множества `seen_ids` могут содержать тысячи элементов и не очищаются после использования.

**Решение:** Очищать после использования или использовать более эффективные структуры данных.

---

### 13. Глобальный HTTP клиент не имеет ограничений на пул соединений
**Файл:** `api/http_client.py:74, 143`
**Проблема:** `pool_size=100` означает, что может быть до 100 открытых соединений одновременно, которые могут накапливаться.

**Решение:** Добавить таймауты для соединений, периодическую очистку неиспользуемых соединений.

---

### 14. Логирование может накапливать данные в памяти
**Файл:** `api/http_client.py:254-263`
**Проблема:** Структурированное логирование может накапливать данные в буферах, особенно при большом количестве запросов.

**Решение:** Использовать ротацию логов, ограничить размер буферов.

---

### 15. Pydantic модели могут создавать циклические ссылки
**Файлы:** Все файлы с моделями
**Проблема:** Pydantic модели могут создавать циклические ссылки, которые не освобождаются сборщиком мусора.

**Решение:** Явно разрывать ссылки после использования.

---

### 16. Retry механизм может создавать множественные запросы
**Файл:** `api/http_client.py:225-298`
**Проблема:** При retry создаются новые запросы, которые могут накапливаться в памяти до завершения.

**Решение:** Ограничить количество одновременных retry, использовать очереди.

---

### 17. Throttling может накапливать задачи в очереди
**Файл:** `api/throttling.py`
**Проблема:** ResourceManager может накапливать задачи в очереди при превышении лимитов.

**Решение:** Ограничить размер очереди, отбрасывать старые задачи.

---

### 18. Глобальные экземпляры парсеров могут удерживать драйверы
**Файл:** `api/che168/parser.py:33`
**Проблема:** `Che168Parser` имеет поле `self.driver`, которое может удерживать WebDriver между вызовами.

**Код:**
```python
self.driver = None
```

**Решение:** Гарантировать закрытие драйвера после каждого использования.

---

### 19. Конвертеры могут создавать временные объекты
**Файлы:** `converters.py`
**Проблема:** Функции декодирования могут создавать временные строки и объекты, которые накапливаются.

**Решение:** Использовать более эффективные методы работы со строками.

---

### 20. Async tasks могут накапливаться
**Файл:** `async_api_server.py:1060, 1198`
**Проблема:** `asyncio.create_task()` создает задачи, которые могут накапливаться, если не отслеживаются.

**Код:**
```python
asyncio.create_task(task_service.process_task(task.id))
```

**Решение:** Отслеживать задачи, ограничивать количество одновременных задач.

---
